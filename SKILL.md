# Cognitive Load Theory: Learning Material Design

A principled framework for designing, reviewing, and refactoring instructional content. Distilled from 30+ years of educational psychology research (Sweller 1988; Sweller, van Merriënboer & Paas 1998; van Merriënboer & Sweller 2005; de Jong 2010; Paas & van Merriënboer 2020).

## Important caveat — read first

CLT was developed and validated for **human classroom and lab learning over short-to-medium time horizons**. Applying it to material generated by an AI agent is a defensible but **unvalidated extrapolation from primary research**. Treat this skill as a structured heuristic, not a guaranteed optimization. When you produce learning material under this skill, briefly state which CLT principles you applied so the human can audit the design choice.

## Core architecture (the only theory you must keep in mind)

- **Working memory (WM)** holds roughly four novel elements for ~20 seconds. It is the bottleneck for all conscious learning.
- **Long-term memory (LTM)** is effectively unlimited and stores **schemas** — chunked patterns that compress many elements into one.
- **Learning = building schemas in LTM by processing information in WM.**
- **Schemas, once automated, bypass WM** and let an expert handle in one chunk what overwhelms a novice.

Every design decision should ask: *Does this respect WM limits and accelerate schema construction?*

## Three types of cognitive load

| Type | Source | Lever |
|---|---|---|
| **Intrinsic** | Material's element interactivity × learner's prior knowledge | Sequence isolated → interacting elements; pre-train on prerequisites |
| **Extraneous** | Poor design — split attention, redundancy, weak search strategies | Eliminate. This is CLT's main attack surface. |
| **Germane** | Effortful cognitive processing that builds/refines schemas | Promote via worked examples, self-explanation, variability |

The three are additive but not cleanly separable. See [references/caveats.md](references/caveats.md) for de Jong's critique that this distinction is hard to test empirically.

## Decision flow when this skill is loaded

### Step 1 — classify the user's request

| Task type | Example phrasing | Primary references |
|---|---|---|
| **GENERATE** | "Make study notes for X", "create a worked example for Y" | effects-catalog, decision-trees |
| **REVIEW** | "Is this teaching material good?", "review this chapter" | review-checklist, effects-catalog |
| **REFACTOR** | "Rewrite this for novices", "simplify this section" | effects-catalog, decision-trees |
| **DIAGNOSE** | "Why does this feel overwhelming?", "I can't follow this" | three-loads explanation in this file, effects-catalog |

### Step 2 — establish learner context

Before applying any effect, determine (or ask):

1. **Expertise level** in this domain:
   - **Novice**: no relevant schemas; will use slow general strategies
   - **Intermediate**: some schemas; combining and extending them
   - **Expert**: rich automated schemas; doing the task fluently

2. **Element interactivity** of the material:
   - **Low**: items learnable in isolation (vocabulary, isolated facts, names)
   - **High**: items must be processed simultaneously (a system's behavior, multi-step procedures, interacting concepts)

These two axes drive everything. The same material needs different treatment for different learners — see the **expertise reversal effect** in `effects-catalog.md`.

### Step 3 — pick effects

Open [references/decision-trees.md](references/decision-trees.md) for the matrix of expertise × intrinsic load → recommended effects.

A compressed version:

- **Novice + high intrinsic load**: worked examples; integrated diagram+text; audio narration over written redescription; eliminate redundancy; sequence isolated → interacting.
- **Intermediate**: completion problems (partial solutions); self-explanation prompts; variable surface stories.
- **Expert**: pure problems; remove now-redundant explanatory text; high contextual interference (random practice schedule).
- **Always**: reduce extraneous decoration; chunk; signal structure clearly.

For each effect's mechanism, conditions, and worked examples, see [references/effects-catalog.md](references/effects-catalog.md).

### Step 4 — produce output

For **GENERATE / REFACTOR**:
1. State assumed learner level and intrinsic load level.
2. List effects you applied (e.g., "worked-example, integrated split-attention, modality").
3. Produce the material.
4. (Optional) Suggest a follow-up step matching guidance-fading: e.g., "after this worked example, try the completion problem in §X."

For **REVIEW**:
1. Identify which effects' principles are violated.
2. Rate each issue's severity by impact on schema construction (high / medium / low).
3. Provide concrete revisions, not vague advice.
Use [references/review-checklist.md](references/review-checklist.md) as a structured pass.

For **DIAGNOSE**:
1. Estimate which load type is dominant (intrinsic vs extraneous).
2. Name the specific mechanism (e.g., "the diagram and text are spatially split, forcing mental integration on top of high intrinsic load").
3. Suggest one concrete change.

## Anti-patterns to avoid

- **"Comprehensive" notes that ignore WM limits** — dumping everything is not teaching.
- **Diagram on one page, explanation on another** (split attention).
- **Long prose narrating a self-explanatory diagram** (redundancy).
- **Same material structure for novice and expert** (expertise reversal).
- **Pure discovery learning for novices** (means-ends search depletes WM, blocks schema building).
- **Post-hoc claim that "this generates germane load"** — per de Jong, this is unfalsifiable. Describe the design choice, not the supposed mental process.

## When NOT to apply this skill

- The user wants information retrieval, not learning (e.g., "look up X for me").
- Content is non-pedagogical: code documentation for engineers who already know the codebase, marketing copy, casual chat, executive summaries.
- The user explicitly wants raw or unfiltered material.
- The user is editing prose, not designing instruction.

## Learner-side strategies (use cautiously)

Some 2020-era CLT research recommends learner-side moves: collaboration, gesturing, eye closure, expressive writing before exams. These are **unvalidated for AI-mediated learning** and target the human, not the material. If you mention them, see [references/learner-side-strategies.md](references/learner-side-strategies.md) and explicitly mark them as research-stage suggestions.
